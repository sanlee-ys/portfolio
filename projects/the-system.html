The System

Four independent repositories, engineered to operate as one system — without collapsing into a monolith.

The interesting work in a multi-project system isn't the projects — it's the seams between them. This is a system of four repos that talk to each other over frozen contracts, run an event-driven classification loop, and stay independently deployable. The deliverable isn't just that it works; it's that every cross-cutting decision is written down and reviewable.

The problem

Four useful-on-their-own projects needed to behave as one product:

  • notes-apiSpring Boot REST service — the knowledge base and data layer.
  • defense-news-classifierLLM classifier with a real eval harness — labels content.
  • kb-agentRAG + tool-use agent — the hub that reasons over the system.
  • learning-notesPlain-language notes on the techniques behind it all.

The easy path is to fuse them into one codebase so they can call each other directly. That buys convenience and pays for it with independence: one release cycle, one blast radius, one thing to break. The harder, better goal was to let them act as one system while each stays deployable on its own. That's an architecture problem, not a coding problem — it lives in the contracts between the pieces.

The decisions

Decision · SYS-004 / SYS-006

Decouple with contracts, not shared code

The repos talk over frozen HTTP and event contracts with explicit versioning rules; both sides carry contract tests in CI.

Why: "decoupling" without a contract is just hope — a renamed field breaks the consumer silently at runtime. Tradeoff: more upfront ceremony (freeze the wire shape, write the tests) so that drift fails a build instead of production.

Decision · SYS-005

Make the loop event-driven and idempotent by design

Creating a note emits a Kafka event; the classifier consumes it, classifies in-process, and writes labels back as namespaced tags (category: / domain:) with replace semantics.

Why: Kafka's at-least-once delivery means events arrive twice, so the writeback has to converge on reprocessing, not accumulate. Note creation never waits on classifier uptime. Tradeoff: idempotency is real design work (independent retry paths, poison-message handling) versus naive fire-and-forget.

Decision · SYS-002

Gate model choice on evidence, not vibes

Default to Sonnet across the system; escalate to Opus only where an eval shows the quality gain pays.

Why: the classifier's accuracy ceiling turned out to be label ambiguity, not model horsepower — paying ~1.7× for a bigger model would have bought nothing. Tradeoff: you have to run the eval to justify the tier, instead of just defaulting to the biggest model.

Decision · SYS-001 / SYS-008

Record decisions, then make the system legible

Cross-repo decisions live as two-tier ADRs (repo-local stays local; system decisions are SYS-NNN in one place). A generated portal aggregates every repo's docs into one browsable view, deep-linking to code — aggregate, never duplicate.

Why: a system you can't see in one place is narrated, not legible — and a decision that isn't written down gets re-argued.

How it flows

  1. A note is created in notes-api, which publishes a NoteCreated event to the Kafka note-events topic.
  2. The classifier's consumer reads the event.
  3. It classifies the text in-process — one Sonnet call, structured output forced via tool-use — into a category and an operational domain.
  4. It writes the labels back as namespaced tags (PUT /notes/{id}/tags, replace semantics), preserving the user's own tags.
  5. The kb-agent reads notes (GET /notes) to ground its RAG answers, and can also call the classifier synchronously.

The outcome

  • Create a note → it's automatically classified and tagged, asynchronously, with no coupling between note creation and classifier availability.
  • Four repos stay independently deployable, and cross-repo drift is caught by a red build — not a production incident.
  • The whole system is browsable in one portal, with links straight to the real code.
  • It's articulated, not just built: a skill map (SYS-007) names the AI-era capabilities the system exercises — evals, context engineering, agents, observability — and the honest gap (security on the agent's tool seam).

What it demonstrates: systems thinking. The value lives in the seams and the decisions — chosen deliberately, recorded so they're reviewable, and protected so they don't silently rot.
